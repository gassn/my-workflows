# Reactでkey propはなぜ必要なのか

## 結論

`key` propは、Reactがリストをレンダリングする際に **「どの要素が同じで、どの要素が変わったか」を正しく識別するため** に必要です。

## 詳しい理由

### 1. Reactの差分検出（Reconciliation）の仕組み

Reactは再レンダリング時、前回のVirtual DOMと今回のVirtual DOMを比較（diff）して、実DOMへの変更を最小限にします。

リストの場合、`key`がないとReactは **「インデックス（位置）」で要素を比較** します。これが問題を引き起こします。

### 2. keyがない場合の問題

例えば以下のようなリストがあるとします。

```jsx
// 初期状態
<ul>
  <li>リンゴ</li>   // index 0
  <li>バナナ</li>   // index 1
  <li>みかん</li>   // index 2
</ul>

// 先頭に「ぶどう」を追加
<ul>
  <li>ぶどう</li>   // index 0
  <li>リンゴ</li>   // index 1
  <li>バナナ</li>   // index 2
  <li>みかん</li>   // index 3
</ul>
```

keyがないと、Reactは以下のように判断してしまいます。

- index 0: 「リンゴ」→「ぶどう」に **変更**
- index 1: 「バナナ」→「リンゴ」に **変更**
- index 2: 「みかん」→「バナナ」に **変更**
- index 3: 新規追加「みかん」

本来なら「先頭に1つ追加しただけ」なのに、すべての要素が更新されてしまうのです。

### 3. keyがあると何が起こるか

```jsx
<li key="apple">リンゴ</li>
<li key="banana">バナナ</li>
<li key="orange">みかん</li>
```

Reactはkeyを使って「リンゴ」「バナナ」「みかん」がそのまま存在することを認識し、「ぶどう」だけを新規追加と判断できます。 → **差分更新が最小限になり、パフォーマンスが向上**。

### 4. さらに重要: コンポーネントの状態保持

これがkeyの **本当に重要な役割** です。

```jsx
{items.map(item => (
  <InputField key={item.id} defaultValue={item.name} />
))}
```

`<InputField>` が内部にstate（例: ユーザーが入力中のテキスト）を持つ場合、keyが正しくないと、リスト並び替え時に **状態が別の要素に引き継がれてしまう** バグが発生します。

keyが正しければ、Reactは「このコンポーネントインスタンスはこのkeyのやつ」と追跡できるので、状態が正しい要素に紐づきます。

## なぜ`index`をkeyにするのは非推奨なのか

```jsx
items.map((item, index) => <li key={index}>{item}</li>)  // アンチパターン
```

- リストの順序が変わる・追加/削除があると、同じ要素にも違うindexが割り当たる
- 結果的にkeyなしと同じ問題が発生する

**リストが静的で並び替えも追加削除もない場合のみ** 、indexでも許容されます。

## まとめ

| 役割 | 説明 |
|------|------|
| **識別子** | どの要素が同一かをReactに教える |
| **パフォーマンス** | 差分更新を最小限にする |
| **状態保持** | コンポーネントのstateを正しい要素に紐づける |

keyは「一意で」「安定していて」「予測可能」な値（通常はDBのIDなど）を使うのがベストプラクティスです。
