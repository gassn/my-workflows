# Spec: login

## 1. 目的

Web アプリケーションに基本的なログイン機能を追加し、ユーザー自身のアカウントで保護領域 (`/dashboard`) にアクセスできるようにします。現状は全ページがパブリックであり、個別の設定やユーザー固有のデータを持てない状態を解消することが狙いです。対象利用者は一般のアプリ利用者本人であり、管理者向け機能は本 Spec のスコープ外です。

## 2. スコープ

### 2.1 含むもの

- サインアップ機能 (メールアドレス + パスワード)
- ログイン機能 (メールアドレス + パスワード照合)
- ログアウト機能 (セッション破棄)
- セッション管理 (Cookie ベース)
- 認証失敗時のエラーメッセージ表示
- 保護ページ (`/dashboard`) へのアクセス制御
- パスワードの bcrypt ハッシュ化保存

### 2.2 含まないもの

- パスワードリセット機能
- ソーシャルログイン (Google / GitHub 等)
- 2 要素認証 (2FA / MFA)
- メールアドレス確認フロー (認証メール送信)
- 管理者向け機能 (ユーザー管理画面等)
- 新規認証プロバイダ (Auth0 等) の導入

## 3. 機能要件

### 3.1 サインアップ

新規ユーザーがメールアドレスとパスワードでアカウントを作成します。

**入力**:
- `email`: 文字列。RFC 5322 準拠のメールアドレス形式。最大 254 文字。一意制約 (既存ユーザーと重複不可)。
- `password`: 文字列。8 文字以上、128 文字以下。

**処理**:
- パスワードは bcrypt (cost = 12) でハッシュ化し、`users` テーブルに `email`, `password_hash`, `created_at` として保存します。
- 保存後、自動でログイン状態にします (セッション発行)。

**出力**:
- 成功時: `/dashboard` へリダイレクト (HTTP 302) + セッション Cookie 発行。
- 失敗時: サインアップ画面を再表示し、エラーメッセージを含めます。

**エラーハンドリング**:
- メール形式不正: 「メールアドレスの形式が正しくありません」
- パスワード長不足 / 超過: 「パスワードは 8 文字以上 128 文字以下で入力してください」
- メール重複: 「このメールアドレスは既に登録されています」
- DB エラー: HTTP 500 + 一般的なエラーメッセージ「しばらく経ってから再度お試しください」

### 3.2 ログイン

既存ユーザーがメールアドレスとパスワードで認証します。

**入力**:
- `email`: 文字列。
- `password`: 文字列。

**処理**:
- `users` テーブルから `email` で検索し、`bcrypt.compare(password, password_hash)` で照合します。
- 照合成功時、新規セッション ID を発行し、サーバ側セッションストア (PostgreSQL `sessions` テーブル) に保存します。
- セッション Cookie を発行します (属性は下記)。

**出力**:
- 成功時: `/dashboard` へリダイレクト (HTTP 302) + セッション Cookie 発行。
- 失敗時: ログイン画面を再表示し、汎用エラーメッセージを表示。

**エラーハンドリング**:
- 認証情報の不一致 (メール不存在 / パスワード不一致の両ケース): 同一のエラーメッセージ「メールアドレスまたはパスワードが正しくありません」を返し、どちらが不一致かは判別不可とします (ユーザー列挙攻撃対策)。
- 入力欠落: 「メールアドレスとパスワードを入力してください」

### 3.3 ログアウト

ログイン中のユーザーが明示的にセッションを終了します。

**入力**:
- セッション Cookie (リクエストヘッダ)。

**処理**:
- サーバ側セッションストアから該当セッション ID のレコードを削除します。
- レスポンスで Set-Cookie により Cookie を期限切れにします (`Max-Age=0`)。

**出力**:
- `/login` へリダイレクト (HTTP 302)。ログイン画面へ誘導する方が未ログイン状態のユーザー体験上自然なためです。

**エラーハンドリング**:
- 無効 / 期限切れセッションでのログアウト要求: 冪等に扱い、200 相当で `/login` にリダイレクトします (エラー表示しません)。

### 3.4 セッション管理

ログイン状態を Cookie ベースで維持します。

**セッション Cookie 属性**:
- `HttpOnly`: true (JavaScript からアクセス不可)
- `Secure`: true (HTTPS 接続時のみ送信)
- `SameSite`: `Lax` (CSRF 緩和)
- `Path`: `/`
- `Max-Age`: 86400 秒 (1 日)

**セッション有効期限**: 1 日 (24 時間)。利用頻度が日次であると想定し、長すぎず短すぎない値を採用します。

**セッションストア**: PostgreSQL の `sessions` テーブル (カラム: `id`, `user_id`, `created_at`, `expires_at`)。`expires_at` 経過済みレコードは参照時に無効扱いとし、日次バッチで物理削除します (日次削除は本 Spec のスコープ外、運用タスクとして後続整備)。

### 3.5 保護ページのアクセス制御

`/dashboard` 及び将来追加される保護ページへのアクセスはセッション有効なユーザーのみ許可します。

**処理**:
- Express middleware で全 `/dashboard/*` リクエストのセッション Cookie を検証します。
- 有効セッションがない場合、`/login?redirect=<元 URL>` にリダイレクト (HTTP 302) します。

**エラーハンドリング**:
- セッション期限切れ: 同上、`/login?redirect=...` へリダイレクト。
- セッション改ざん (不正な ID): 同上、Cookie 削除 + `/login` リダイレクト。

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | ログイン API の応答時間は P95 で 500ms 以内 (bcrypt cost = 12 想定、標準的なクラウド VM 環境で検証)。サインアップも同様。 |
| セキュリティ | パスワードは bcrypt (cost = 12) でハッシュ化し、平文保存を禁止。Cookie は HttpOnly / Secure / SameSite=Lax。CSRF トークンはフォーム送信系エンドポイント (ログイン / サインアップ / ログアウト) で必須。ユーザー列挙防止のため認証失敗メッセージは汎用化。 |
| 可用性 | 既存 Express + PostgreSQL 構成の可用性に準拠 (本 Spec では新規 SLA は設定しない)。PostgreSQL ダウン時は HTTP 503 を返却。 |
| 拡張性 | 将来のパスワードリセット / 2FA / ソーシャルログイン追加時に `users` テーブル / セッション管理機構を拡張可能な構造とする (例: `users.password_hash` カラム名ではなく `users.auth_credential` のように将来の認証方式追加を見越した命名にするかは実装判断)。 |
| 保守性 | 認証関連の全アクション (サインアップ / ログイン / ログアウト / 認証失敗) は構造化ログ (JSON 形式) に記録する。ログには `user_id` (既知の場合)、`event_type`、`timestamp`、`ip_address`、`user_agent` を含める。パスワード・セッション ID はログに記録しない。 |

## 5. 受け入れ基準 (Acceptance Criteria)

- [ ] 未登録ユーザーがメールアドレスとパスワード (8 文字以上) を入力してサインアップでき、DB に bcrypt ハッシュ化された状態で保存されることを確認できる。
- [ ] 正しいメールアドレスとパスワードでログインすると HTTP 302 で `/dashboard` にリダイレクトされ、セッション Cookie (HttpOnly / Secure / SameSite=Lax) が発行されることを確認できる。
- [ ] 誤ったパスワード、または存在しないメールアドレスでログインを試みた場合、画面に「メールアドレスまたはパスワードが正しくありません」という汎用メッセージが表示され、どちらが不一致かは判別不能である。
- [ ] 未ログイン状態で `/dashboard` にアクセスすると `/login?redirect=/dashboard` にリダイレクトされる。
- [ ] ログイン後、ログアウト操作を実行するとセッション Cookie が無効化され (Max-Age=0)、サーバ側セッションストアから該当レコードが削除され、`/login` にリダイレクトされる。
- [ ] セッション有効期限 (24 時間) 経過後、保護ページにアクセスすると `/login?redirect=...` にリダイレクトされる。
- [ ] サインアップで重複メールアドレスを送信すると「このメールアドレスは既に登録されています」が表示される。
- [ ] サインアップで 7 文字以下または 129 文字以上のパスワードを送信するとエラーメッセージが表示され、DB に保存されない。
- [ ] ログイン API の応答時間が標準検証環境で P95 500ms 以内である (bcrypt cost = 12)。
- [ ] 認証関連の全アクションが構造化ログ (JSON) に記録され、パスワードやセッション ID が含まれていないことを確認できる。

## 6. 非対象 (スコープ外)

- **パスワードリセット機能**: 時間的制約 (2 週間) により初回リリース対象外。次期 Spec で別途対応します。
- **ソーシャルログイン (Google / GitHub 等)**: 「既存構成維持」「新規認証プロバイダ非導入」の制約により除外します。
- **2 要素認証 (2FA / MFA)**: 初回リリースの単純化のため除外します。セキュリティ強化要件が発生した段階で別 Spec として検討します。
- **メールアドレス確認フロー**: メール送信基盤が未整備のため除外します。メール基盤整備と併せて後続 Spec で対応します。
- **管理者機能**: 「対象利用者は一般ユーザーのみ」の方針により除外します。
- **レート制限の本格実装**: 本 Spec ではリスク 7.2 の緩和策として簡易 IP ベースの試行回数制限のみ導入し、本格的な WAF / 分散レート制限は別 Spec で扱います。
- **セッション失効済みレコードの物理削除バッチ**: 運用タスクとして別途整備します。論理的な失効 (参照時の `expires_at` 確認) のみ本 Spec に含みます。

## 7. リスクと緩和策

### 7.1 セッションハイジャック

- **内容**: Cookie 盗難 (XSS、中間者攻撃、物理アクセス等) により攻撃者が被害者のセッション Cookie を取得し、不正ログイン状態でアプリにアクセスするリスク。
- **緩和策**:
  - Cookie に `HttpOnly` / `Secure` / `SameSite=Lax` 属性を必ず付与する。
  - HTTPS を本番環境で強制する (既存インフラ設定に依存、本 Spec では確認のみ)。
  - セッション ID はサーバ側で暗号論的に安全な乱数 (128 bit 以上) で生成する。
  - セッション有効期限を 24 時間に設定し、被害時間を限定する。
  - ログイン成功時、既存セッションがあればセッション ID を再生成する (セッション固定攻撃対策)。

### 7.2 ブルートフォース攻撃

- **内容**: 攻撃者が同一メールアドレスに対してパスワードを総当たりで試行し、弱いパスワードのアカウントを突破するリスク。
- **緩和策**:
  - IP ベースの簡易レート制限を導入する: 同一 IP からのログイン試行を 1 分間に 10 回までに制限 (Express middleware で実装、PostgreSQL 側で試行履歴を保持)。超過時は HTTP 429 を返却。
  - ログイン失敗ログを構造化ログに記録し、監視基盤側での検知余地を残す。
  - パスワード最小長 8 文字を強制し、総当たり空間を一定以上に保つ。
  - 本格的な分散レート制限 / アカウントロックは別 Spec で扱います (スコープ外)。

### 7.3 bcrypt コスト設定のトレードオフ

- **内容**: bcrypt のコストパラメータが過大だと応答遅延・サーバ負荷が増大し、過小だと将来的なハードウェア進化で解析可能になるリスク。
- **緩和策**:
  - 初期値として cost = 12 を採用 (2026 年時点の一般的推奨値)。
  - 非機能要件の応答時間 (P95 500ms) を基準に計測し、閾値超過時はコストを再検討する。
  - コストはアプリ設定 (環境変数) から変更可能にし、将来ハードウェア性能向上に応じて増加できるようにする。
  - ログイン応答時間を監視対象に含める。

### 7.4 CSRF 攻撃

- **内容**: ログイン済みユーザーを悪意あるサイトから誘導し、ログアウト・退会等の状態変更操作を勝手に実行させるリスク。
- **緩和策**:
  - 全フォーム送信系エンドポイント (ログイン / サインアップ / ログアウト) に CSRF トークンを必須化する。
  - Cookie の `SameSite=Lax` 属性によりクロスサイト POST を抑止する。

### 7.5 ユーザー列挙攻撃

- **内容**: ログイン失敗時のエラーメッセージを「メールアドレスが存在しない」「パスワードが違う」と区別すると、攻撃者が登録済みメールアドレスを列挙できてしまうリスク。
- **緩和策**:
  - 認証失敗時のエラーメッセージを「メールアドレスまたはパスワードが正しくありません」に統一する (機能要件 3.2 参照)。
  - サインアップ時のメール重複エラーは業務上の利便性を優先して表示するが、レスポンス時間をメール不存在ケースと揃えるようダミー処理 (bcrypt.compare を常に実行する等) を検討する (実装時判断)。
