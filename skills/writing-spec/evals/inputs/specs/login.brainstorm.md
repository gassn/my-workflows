# Brainstorming: login

## 目的
Web アプリケーションに基本的なログイン機能を追加し、ユーザー自身のアカウントで保護領域にアクセスできるようにします。現状は全ページがパブリックで、個別の設定やデータを持てない状態を解消することが狙いです。

## 利用者
一般ユーザー (アプリ利用者本人)。管理者機能は対象外。

## 成功条件 (受け入れ基準)
- メールアドレスとパスワードでログインできる
- ログイン成功後、保護ページ (/dashboard) にアクセスできる
- 誤った認証情報でログインを試みた場合、エラーメッセージが表示される
- ログイン状態はセッション Cookie で維持される
- ログアウト操作でセッションが破棄される

## 制約
- **技術的制約**: 既存の Express + PostgreSQL 構成を維持する。新規認証プロバイダ (Auth0 等) は導入しない
- **セキュリティ制約**: パスワードは bcrypt でハッシュ化して保存する (平文禁止)
- **時間的制約**: 2 週間以内にリリース

## スコープ
### 含むもの
- サインアップ (メール + パスワード)
- ログイン / ログアウト
- セッション管理 (Cookie)
- 認証失敗時のエラー表示

### 含まないもの (明示的に除外)
- パスワードリセット機能
- ソーシャルログイン (Google / GitHub 等)
- 2 要素認証
- メールアドレス確認フロー (認証メール送信)

## 代替案の検討
- **Auth0 等の外部認証サービス**: 却下。既存構成維持の制約あり
- **セッション管理方式**: Cookie セッション vs JWT。Cookie セッションを採用 (サーバ側破棄が容易)

## リスク
- **セッションハイジャック**: Cookie 盗難による不正ログイン。HttpOnly / Secure フラグで緩和
- **ブルートフォース攻撃**: パスワード総当たり攻撃。レート制限の要否を Spec で判断
- **bcrypt コスト設定**: コスト過大で応答遅延、過小で脆弱性。推奨値 (12) で開始

## 未解決事項
- セッション有効期限 (1 時間 / 1 日 / 1 週間のいずれか)
- レート制限の導入要否 (Spec で決定)
- ログアウト時のリダイレクト先 (/ か /login か)
- エラーメッセージの粒度 (「メールまたはパスワードが不正」の汎用か、詳細化するか)
