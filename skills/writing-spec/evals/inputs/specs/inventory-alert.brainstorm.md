# Brainstorming: inventory-alert

## 目的
EC サイトの管理画面で在庫切れを未然に防ぐため、在庫数が一定水準を下回った時点で管理者へ自動的にメール通知を行います。手動での在庫監視を不要にし、品切れ機会損失を低減することが狙いです。

## 利用者
管理者 1 名 (利用者本人) のみ。チーム運用や外部公開は想定しません。

## 成功条件 (受け入れ基準)
- 商品の在庫数が 10 個未満になった時点で、管理者宛にメール通知が送信される
- メール送信処理は既存のメール送信モジュールを再利用して実装される
- 来月のリリースまでに本機能を稼働可能な状態にする
- SMS 通知 / Slack 連携が含まれていないこと (スコープ厳守)

## 制約
- **時間的制約**: 来月リリース予定 (具体的な日付は Spec フェーズで確定)
- **技術的制約**: 既存のメール送信モジュールを再利用する (新規メール基盤の構築は不可)
- **既存システムへの影響**: 在庫数の変動を検知する仕組みが必要 (既存の在庫更新ロジックへフックを追加するか、定期バッチで監視するかは Spec フェーズで決定)

## スコープ
### 含むもの
- 在庫数 10 個未満の検知ロジック
- 既存メール送信モジュールを呼び出す通知処理
- 管理者宛通知メール (宛先は管理者本人のメールアドレス)

### 含まないもの (明示的に除外)
- SMS 通知
- Slack 連携
- 通知チャネルの増設・設定 UI
- 複数管理者への配信、ロール別通知

## 代替案の検討
- **SMS 通知 / Slack 連携**: ユーザーが明示的に除外。チャネル追加は将来検討事項として保留
- **検知方式 (リアルタイム vs バッチ)**: ヒアリング段階では未確定。Spec フェーズで以下のトレードオフを比較する
  - リアルタイム (在庫更新時フック): 即時性が高いが既存在庫更新フローへの影響が発生
  - 定期バッチ: 既存ロジックへの侵襲が小さいが通知遅延が発生

## リスク
- **通知の重複送信**: 在庫が閾値付近で増減を繰り返した場合、同一商品で大量のメールが送信される懸念。デバウンス / 既送信フラグ等の制御方針を Spec フェーズで決定する必要あり
- **メール未達**: 既存メール送信モジュールの障害時に通知漏れが起き、在庫切れに気付けない可能性。再送やフォールバックの要否を Spec フェーズで議論
- **閾値 (10 個) の妥当性**: 商品ごとに適切な閾値が異なる可能性 (回転の速い商品は 10 個では遅いなど)。今回は固定値で開始するが、将来的な可変化のための拡張余地を考慮する

## 未解決事項
- 在庫検知方式 (リアルタイムフック / 定期バッチ) の選定
- 通知メールの宛先メールアドレス (環境変数 / 設定ファイル / DB)
- 通知メールの本文テンプレート (商品名、SKU、現在在庫数など含めるべき情報)
- 重複通知の抑制ポリシー (1 商品 1 日 1 通など)
- 「10 個未満」の境界条件 (10 を含むか含まないか / `<` か `<=` か)
- 来月リリースの具体的な締め切り日
- 既存メール送信モジュールのインターフェース仕様 (Spec フェーズで実装調査)
- 在庫データの参照元テーブル / API

## ヒアリングログ (任意)
- ユーザー初回発話で目的・利用者・成功条件・制約・スコープ・除外項目がすべて提示された
- ユーザーから「Spec フェーズに進んで良い」と明示的な移行承認を取得済み
- 代替案・リスクは未言及だったため、Spec フェーズで議論する事項として「未解決事項」セクションに整理
